## Main Finding
The current full e2e suite is not a clean signal for the current app. It mixes real app coverage with stale selectors, local-only QA flags, mutable CMS seed assumptions, and a fragile authenticated harness. The authenticated failures should not be interpreted as independent app regressions until auth/session setup is fixed.

## Authentication Finding
Production-like tests already use test-login in global setup: `tests/helpers/global-setup.ts:84`. The problem is reliability and scope, not absence. A single shared `test-results/playwright/admin-storage-state.json` is reused by the authenticated project, while fixture helpers call admin APIs through the Playwright `request` fixture. The prior full run still produced admin redirects to `/login` and 401 fixture creation errors.

Required direction: keep test-login for production-like tests, but make it explicit and fresh in the e2e harness. Add an authenticated API request context for fixture creation and refresh admin UI sessions before mutation-heavy admin flows.

## Test Currency Findings
- Outdated detail title selectors: `tests/e2e-visitor-content-exploration.spec.ts:16`, `tests/e2e-visitor-content-exploration.spec.ts:37`, `tests/public-detail-pages.spec.ts:18`, and `tests/public-detail-pages.spec.ts:43` use `main h1` as if it were unique. Current work/blog detail pages intentionally render body h1 headings inside `#work-detail-content` and `#blog-detail-content`, so these tests should scope to the article title.
- Local QA flag mismatch: `src/components/content/LocalQaQueryBoundary.tsx:5` only accepts `localhost` and `127.0.0.1`. Production-like internal Docker hostnames do not trigger `__qaBroken` or `__qaEmpty`, so `tests/public-api-error.spec.ts` and `tests/public-resume-empty-state.spec.ts` are environment-invalid as written.
- Mutable seed assumption: tests asserting exact CMS content like `Introduction` are vulnerable to real admin edits in the production-like database. They should create fixture data or assert stable structure.
- Work video tests are mostly aligned with current implementation: `WorkVideoPlayer` still exposes the resize and preview test IDs, and the detail page renders it for uploaded videos. The full run failed before meaningful video assertions because admin creation/auth failed.
- Notion tests are also mostly blocked rather than clearly stale: current Notion workspace still exposes the tested triggers, but autosave specs failed during fixture/admin setup.
- Recording/batch suites are a separate lane concern. Specs depending on `window.__WOONG_TIPTAP_EDITOR__` and large recording fixtures should not block the core production-like lane until they are updated.

## Recommended Stabilization Order
1. Fix e2e auth harness first: authenticated API fixture context, fresh admin session for admin UI flows, and isolation for logout/session mutation tests.
2. Update stale public selectors and QA flag environment assumptions.
3. Replace mutable CMS seed assertions with fixture-backed or structural assertions.
4. Rerun targeted groups: public detail/TOC, admin smoke, Notion autosave, works video upload/preview.
5. Keep recording/live/batch visual suites optional until their instrumentation and setup are current.

## Final Recommendation
Do not treat the current 209 failures as 209 app regressions. The suite is partially out of sync with the current code and partially blocked by auth/session fixture design. Stabilize auth and stale selectors before using full-suite timing as a reliable production-like signal.
