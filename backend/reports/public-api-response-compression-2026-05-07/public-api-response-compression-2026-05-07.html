Date: 2026-05-07

Summary

ASP.NET Core response compression is enabled for JSON/problem+JSON/text responses. The public Work detail endpoint now honors Accept-Encoding: gzip and returns Content-Encoding: gzip with Vary: Accept-Encoding.

Changed

  • Registered response compression, gzip provider, HTTPS compression, and JSON MIME types.
  • Added app.UseResponseCompression() to startup order.
  • Added behavior-focused public endpoint compression coverage.
  • Updated middleware architecture guard.
  • Stored k6 compression smoke artifact.

Not Changed

  • No cache.
  • No seeded-target forcing.
  • No pageSize=12 change.
  • No DB query or DTO semantic change.
  • No nginx timing extraction change.

Validations

  • Focused compression test: RED before implementation, GREEN after implementation.
  • Architecture middleware order test: passed 1/1.
  • Public endpoint suite: passed 31/31.
  • Full backend solution: passed; Contract 1 skipped, Component 124, Unit 59, Architecture 35, Integration 229.
  • Dev compose backend/nginx rebuild: succeeded.
  • Curl smoke: plain 1195 B, gzip 850 B, expected gzip headers present.
  • k6 smoke: 500 requests, check rate 1.0, HTTP failure rate 0, p95 2.415107 ms.

Risks

Gzip can increase CPU on the 2 CPU shared host. The local seeded-work smoke proves behavior but does not replace a production-like A/B on real heavy detail pages.

Recommendation

Merge if CI is green. Then run a production-like A/B over HTTPS on real heavy Work/Study detail URLs and compare client p95/p99, receiving time, data_received, app elapsed, CPU/GC, DB pressure, ThreadPool queue, and HTTP 5xx.
